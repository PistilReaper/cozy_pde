# 任务4：神经算子PDE智能体
## 赛事背景
在数据驱动的科学发现时代，Cell、Nature、Science（CNS）等顶级期刊不仅是前沿科学的殿堂，更已成为 AI for Science (AI4S) 探索的核心阵地。从蛋白质结构的精准预测到复杂生命系统的模拟，AI 正在重塑基础科学的边界。然而，在传统的科研流程中，AI 往往只作为静态的辅助工具，其能力不仅高度依赖人类科学家的手动调参，且面对 CNS 级别的宏大科学问题时，传统的“人力主导”模式在处理超大规模、高维数据时已触及认知与效率的瓶颈。

自主科研 Agent 的兴起，预示着科研范式的下一次跃迁。本次挑战赛旨在探索 AI 在顶级科学研究中的极限潜力：我们不仅挑战让 Agent 自动化地复现与极致优化 CNS 上的前沿模型，更期望它们能展现出前所未有的自主创新能力，去推理、设计并尝试解决具有 CNS 影响力的重大科学问题。 在这里，AI 将完成从“被设计的模型”到“能解决世界级难题的 AI 科学家”的华丽蜕变。

## Agent 核心任务：构建科研闭环能力
为了应对 CNS 级别的挑战，Agent 必须具备以下四个核心阶段的自动化能力：

- **文献解析与逻辑解构**： Agent 需具备深度的多模态阅读能力，能够自主研读顶刊论文及其配套源代码，精准理清底层数学公式、数据流向、核心算法逻辑以及科学问题的本质。
- **瓶颈诊断与假设提出**： 面对既有模型或未解问题，Agent 需通过分析原始训练日志（Log）或科学数据，敏锐识别在计算效率、模型泛化能力或理论覆盖面上的痛点，并自主提出具有科学依据的优化假设或求解策略。
- **自主设计与代码演进**： 突破”黑盒调用”的限制，Agent 需根据其诊断结果和科学假设，制定具体的工程方案（如：重构网络结构、设计全新的损失函数或引入外部科学工具），并直接编写/修改高质量代码来实现这些创新。
- **实验验证与科学迭代**： Agent 需在测试环境中自主运行干实验，管理多分支的实验版本。通过对实验结果的高效分析，判断当前策略的有效性，并据此调整方向，完成自我进化的科学迭代。

## 赛题背景
偏微分方程（PDE）是刻画复杂动态物理系统的核心数学工具，广泛应用于气候模拟、流体力学、能源工程与材料科学等关键领域。传统数值方法（如有限差分）具有严格的物理一致性与高精度，但其计算代价极高。在高分辨率或多参数场景下，单次模拟可能耗时数周甚至数月，形成显著的算力瓶颈，严重制约大规模科学探索与工业优化。

近年来，科学机器学习（Scientific Machine Learning）为这一难题提供了新的技术路径。其中，神经算子（Neural Operator）通过学习函数空间到函数空间的映射，直接构建“初始条件 → 解函数”的算子近似。一旦训练完成，原本依赖高性能计算集群反复迭代的数值求解过程，可在毫秒级完成推断，实现数量级上的计算加速。与传统神经网络不同，神经算子在算子层面建模连续映射关系，从理论上具备跨分辨率泛化能力。在工业实践中，NVIDIA 主导研发的 FourCastNet 已展示GPU驱动神经算子在大规模气候建模中的巨大潜力；上智院主导研发的伏羲（Fuxi）模型则可实现全球未来 15 天逐小时温度、风速、降水与辐照等要素预测。

从研发路径看，当前神经算子的研究主要沿两条方向演进：从模型架构引导数据驱动出发，DeepONet(NMI2021)强调 branch/trunk 结构对算子学习的通用性与可表达性[1]；FNO(ICLR 2021; JMLR 2023) 将积分核在 Fourier 空间参数化，强调其对分辨率的“离散化一致性”[2]。另一方面，从物理约束角度出发，PI-DeepONet(Science Advance 2021) 基于PINN(Journal of Computaional Physics 2019)提出用自动微分把 PDE 约束加入训练[3][4]。此外，ICON(PNAS 2023)将物理条件与数值解过程引入 in-context learning 框架，展现出跨问题泛化的潜力[5]。尽管上述方法在计算效率或物理一致性方面取得显著进展，但其本质上仍属于“静态模型优化”范式：模型设计、参数调优与验证流程高度依赖人工干预，缺乏系统性的自适应改进机制。在高精尖的应用场景中，效率与精度往往难以同时达到最优平衡。

基于此，我们此次赛题提出一个核心问题：能否构建面向PDE神经算子的科学智能体（AI Agent），使其自动完成“问题理解—模型改进—物理验证—数值复现”的闭环流程，从而在效率与精度之间实现动态优化与持续进化​？

## 赛题介绍
本赛道定位为”AI4S 场景下的科研工作流 Agent 竞赛”：研究对象为神经算子（学习输入函数/系数/初边值 → 解场/时序轨迹 的算子映射），参赛者需开发 LLM Agent 自动完成“理解—改进—验证—复现”的科研闭环，而非仅提交一个训练好的模型权重。

本赛道的核心是从传统的“手动调参”向“自主科研”的范式跃迁。参赛者将不再扮演直接优化神经网络的“做题家”，而是要化身“全流程管理者”，开发一个能够独立接管科研工作流的 PDE Agent。

**构建什么样的 Agent？**
参赛者需要开发一个具备深度逻辑推理、架构演进与复杂代码工程能力的自动化智能体。该 Agent 需突破单纯的”对话交互”，深度整合工具调用（Tool-use）、代码执行沙盒与错误日志解析能力。

**如何运转科研闭环？**
Agent 被唤醒后，需在零人工干预的环境下，自主驱动完整的科学迭代闭环：
- 理解： 研读 PDEBench 数据集规范与主流神经算子（如 FNO、PI-DeepONet）的基线代码。
- 诊断与假设： 分析运行日志，敏锐识别基线模型在”精度”或”长时稳定性”上的物理与数学瓶颈，提出优化假说。
- 代码演进： 自主编写并修改高质量的 Python 代码（如重构网络拓扑、引入物理残差损失函数等）。
- 验证与反思： 运行干实验，像人类程序员一样排查报错（Debug），根据实验结果评估假说，调整方向并记录完整的科研日志。

比赛的目标并非单纯比拼谁能提供一个更优的模型权重，而是评测Agent 本身的“科研探索能力”​。最终，优秀的 Agent 将像经验丰富的科学家一样，通过严密的逻辑推导而非盲目算力试错，自主寻找到突破神经算子长时预测等业界难题的创新方案，并输出极具解释性的Agent 科研实验日志。

### 初赛
本次比赛基于标准化的科学数据集 PDEBench，要求 Agent 针对三大基线模型（FNO、DeepONet、PI-DeepONet）进行自主诊断与演进。比赛分为初赛与复赛两个阶段，难度逐级递增。

初赛使用 PDEBench（DOI: 10.18419/darus-2986） 的 1D 子集，覆盖 1D Burgers，研究重点侧重算法开发/模型优化，规则如下：

- 中长期统计预测​：中长期考察模型的稳定预测能力；
- 物理信息约束​：可利用物理信息参与训练/推理；
- 模型权重微调​：可加载官方 PDEBench checkpoint进行微调。

### 任务：1D Burgers 短窗预测

#### 数据和官方checkpoint说明
- Task-1: 固定物理环境预测
  - 1D_Burgers_Sols_Nu0.001.hdf5是 PDEBench 官方提供的 10000 个样本的 200 时间步长、1024 空间长度的数据集。
  - 官方checkpoint: 1D_Burgers_Sols_Nu0.001_FNO.pt, 1D_Burgers_Sols_Nu0.001_Unet-PF-20.pt。
  - 注意，由于模型在训练时设置的参数中 reduced_resolution_t=5, reduced_resolution=4，所以实际上模型是在 200/5=40 个时间步长，1024/4=256 个空间长度上进行验证，即：模型预测的 1 个时间步，相当于真实数据的 5 个时间步，空间长度同理。
  - 模型官方链接：burgers_FNO.tar 和 burgers_UNET.tar。
- Task-2: 多物理环境预测
  - Task-2 要求模型能够泛化到多种粘性系数（Nu）条件下的 Burgers 方程求解。
  - 训练数据包含多种不同粘性系数的 Burgers 方程样本，并提供每个样本对应的 Nu 值。模型需要学习在不同物理参数下的通用求解能力。
  - 测试数据不提供 Nu 值，仅包含初始条件。选手可自行决定是否在训练过程中利用 Nu 信息，但推理时模型需仅基于初始条件进行预测。
  - 测试集包含 1000 个样本，覆盖多种粘性系数，数据维度为 200 时间步长、256 空间长度。
  - Task-2 不评测训练时间，仅评测预测精度，但推理时间必须在 2 分钟以内，否则该任务得 0 分。

### 新的长时验证数据集 (Validation dataset)
现在，我们扩充了时间步长的数据集，其中包含 1000 个样本的 1000 时间步长、1024 空间长度的数据集。这样就能在 1000/5=200 个步长、256 空间长度的数据中评测模型的性能。

### 评分规则
系统首先通过日志文件，进行规则检测。通过规则检测后，按以下规则计算分数。总分 = Task 1 得分 + Task 2 得分，满分 300 分。

- Task 1 评分（最高 150 分）
  Task 1 总分 = 预测精度得分（最高 75 分）+ 训练耗时得分（最高 45 分）+ 推理耗时得分（最高 30 分）。
  - 预测精度得分 = 分段预测得分 × 0.75。
  - 训练耗时得分（含 Agent 思考时间）：≤60 分钟得 35 分，≤120 分钟得 25 分，≤300 分钟得 20 分，≤500 分钟得 10 分，>500 分钟得 0 分。
  - 推理耗时得分：0 分钟得 40 分，0-2 分钟线性递减，>2 分该任务得 0 分。

- Task 2 评分（最高 150 分）
  Task 2 总分 = 分段预测得分 × 1.5（最高 150 分）。训练时间（包含agent思考时间）不计入评测，但为了防止算力过度消耗，总时长需控制在12小时内。推理时间超过 2 分钟则该任务总分为 0。

- 分段预测得分（Task 1 和 Task 2 通用）
  评分仅针对 190 个预测时间步（去掉前 10 个初始条件）。这 190 步被分为 3 段（T=190，t1=47，t2=95）：
  - 第 1 段（0-47 步，权重 25%）：score = 100 × exp(-20 × Rel-MSE)
  - 第 2 段（47-95 步，权重 25%）：score = 100 × exp(-10 × Rel-MSE)
  - 第 3 段（95-190 步，权重 50%）：取 Lorentzian 分量与 Frechet 分量中的较高者。Lorentzian = 100 / (1 + 10 × RMSE)，Frechet = 50 × exp(-FD²)。
  - 分段预测总分 = 0.25 × score1 + 0.25 × score2 + 0.5 × score3，范围 0-100。

Rel-MSE 计算：逐样本、逐时间步计算 rel_t = Σ(pred-gt)² / Σ(gt²)，对时间步取均值（单样本上限 5.0），再对所有样本取均值。Rel-MSE = 0 为完美预测。

### Log要求
赛事规则中一直明确要求提交内容须完全由 Agent 自主完成，不得人工干预。我们在审查近期提交时，发现部分队伍的提交中存在人工干预 Agent 的痕迹。为便于评审核查，我们对 task1_logs.log 与 task2_logs.log 的格式进行规范，即日起生效。

新的 Log 文件要求

Log 文件的每一行必须是一条合法的 JSON 数据，记录 Agent 每次调用 LLM 的完整 response，且必须包含以下两个字段：

字段	说明
timestamp	本次 LLM 调用完成时的 ISO 8601 时间戳（含时区）
elapsed_seconds	本次 LLM 调用耗时（秒）
如何生成符合要求的 Log

我们提供了一个本地转发代理工具 task_log_sample\openai-log\proxy.py，可自动拦截并记录所有经过的 LLM API 请求，生成符合格式要求的 log 文件。

启动方式：

python proxy.py --port 8080 --target https://api.openai.com --log-dir ./logs
​
将 Agent 的 API base URL 指向 http://localhost:8080，代理即会自动转发请求并写入 log。

代码与 Log 的对应要求

提交的 code/ 目录中的代码必须完全由 Agent 生成，不得包含人工编写或修改的内容
代码内容必须能够与 log 中记录的 LLM 调用历史相互对应，即 log 中应能追溯代码的生成过程
违规判定

以下情形将被判定为违规，该任务得分记为 0 分：

Log 文件不存在，或存在行不是合法 JSON
JSON 行缺少 timestamp 或 elapsed_seconds 字段
Log 最后一条记录与第一条记录的时间差超过 12 小时（视为超时或人工介入）
